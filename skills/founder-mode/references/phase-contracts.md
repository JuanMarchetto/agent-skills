# Phase Contracts Reference

Exact markdown schemas for every artifact that passes between phases. Each phase produces a structured markdown file consumed by the next phase. All artifacts are human-readable, editable, and git-committable.

---

## Phase 0 → Phase 1: Bootstrap Output

**File:** `.founder/phase0-bootstrap.md`

```markdown
# Founder Mode Bootstrap
## Session: {sessionId}
## Mode: new|existing
## Idea: {raw idea text}
## Tech Stack: {auto-detected or user-specified}
## Acceptance Criteria: {from config or empty for Phase 1 to generate}
## Relevant Lessons: {loaded from learn-by-mistake, domain-filtered}
## Codebase Scan: {if existing mode: directory tree, package.json, key files}
```

---

## Phase 1 → Phase 2: Evaluation Output

**File:** `.founder/phase1-eval.md`

```markdown
# Idea Evaluation
## Opportunity Score: X/10
## Go/No-Go: GO|NO-GO
## Market Summary: {2-3 sentences}
## Key Risks: {bulleted list}
## Acceptance Criteria: {extracted or user-provided, numbered list}
## Target User: {one sentence}
## Differentiators: {bulleted list}
## Recommended Tech Stack: {if not specified by user}
```

Phase 1 must produce at least 3 testable acceptance criteria. If the idea is too vague, the evaluator flags this in Key Risks and marks the criteria section as `NEEDS_REFINEMENT`.

### Implementation Summaries

**File:** `.founder/phase4-summaries.md`

Per-chunk summaries used for crash recovery and context continuity. Each chunk summary captures what was built and the interfaces exposed, enabling the next chunk to proceed without re-reading all prior source code.

## Phase 5 → Phase 6: E2E Output

**File:** `.founder/phase5-e2e.md`

```markdown
# E2E Test Results
## Tests Generated: X
## Tests Passing: X/Y
## Coverage Map: {which acceptance criteria are covered}
## Uncovered Criteria: {which acceptance criteria have no E2E test}
## Test Files: {list of created test files}
```

E2E strategy is selected by project type detected in Phase 2 architecture:
- **Web app:** Playwright-style browser tests
- **CLI tool:** Shell script integration tests
- **Library/SDK:** Integration tests exercising the public API
- **Mobile app:** Maestro tests
- **API server:** HTTP request tests (curl/fetch-based)

## Phase 8: GTM Output

Five files in `.founder/phase8-gtm/`. Consumes Phase 1 (market data, differentiators, target user) + Phase 2 (architecture, tech stack) + Phase 4 (what was built).

All GTM artifacts include the header: `<!-- Generated by founder-mode. Edit before publishing. -->`

### README.md

**File:** `.founder/phase8-gtm/README.md`

Sections: what it does, why it exists, install, quick start, architecture overview, contributing. Generated from Phase 1 (target user, differentiators) + Phase 2 (architecture, tech stack) + Phase 4 (what was built).

### PITCH.md

**File:** `.founder/phase8-gtm/PITCH.md`

One-page pitch summary. Sections: problem, solution, target user, key differentiators, traction metrics (test coverage, passing tests), tech stack, what's next.

### TWEET-THREAD.md

**File:** `.founder/phase8-gtm/TWEET-THREAD.md`

4-6 tweet thread: hook, key feature demos, composition/architecture, install CTA. Each tweet marked with character count.

### PRODUCTHUNT.md

**File:** `.founder/phase8-gtm/PRODUCTHUNT.md`

Tagline (60 chars), description (260 chars), 3 key features with one-liners, maker comment draft, suggested categories/topics.

### CHANGELOG.md

**File:** `.founder/phase8-gtm/CHANGELOG.md`

Initial release entry with all features categorized (Added/Changed/Fixed), following Keep a Changelog format.

## Error Artifact

**File:** `.founder/phaseN-error.md` (where N is the failed phase number)

Written when any phase fails to produce its expected artifact (build errors, skill unavailable, API failures).

```markdown
# Phase {N} Error Report
## Phase: {phase name}
## Status: failed
## Error: {failure details — build errors, skill unavailable, API failures}
## Timestamp: {ISO 8601}
```

Phase status is set to `failed` in state.json. The next applicable gate triggers with the error report and presents: `[retry]` (re-run the phase), `[skip]` (skip and continue), or `[stop]` (save and exit).

## State File

**File:** `.founder/state.json`

Tracks pipeline progress for crash recovery and resume.

```json
{
  "sessionId": "fm-20260317-143022",
  "startedAt": "2026-03-17T14:30:22Z",
  "currentPhase": 4,
  "phaseName": "implementation",
  "status": "in_progress",
  "iteration": 2,
  "maxIterations": 3,
  "mode": "new",
  "phases": {
    "0": { "status": "completed", "artifact": ".founder/phase0-bootstrap.md" },
    "1": { "status": "completed", "artifact": ".founder/phase1-eval.md" },
    "2": { "status": "completed", "artifact": ".founder/phase2-arch.md" },
    "3": { "status": "completed", "artifact": ".founder/phase3-plan.md" },
    "4": {
      "status": "in_progress",
      "artifact": ".founder/phase4-impl.md",
      "chunks": {
        "total": 5,
        "completed": 3,
        "current": 4,
        "summaries": ".founder/phase4-summaries.md"
      }
    },
    "5": { "status": "pending" },
    "6": { "status": "pending" },
    "7": { "status": "pending" },
    "8": { "status": "pending" }
  },
  "gateDecisions": {
    "postEval": { "decision": "continue", "timestamp": "2026-03-17T14:35:10Z" },
    "postImplementation": null,
    "postVerification": null
  }
}
```

Valid phase statuses: `pending`, `in_progress`, `completed`, `failed`.

Resume protocol: Read state.json, find `currentPhase` and chunk progress, load summaries for context, continue from current chunk. If a phase is completed but its gate decision is null, re-display the gate rather than re-running the phase.

## Config File

**File:** `.founder/config.json`

Generated during Phase 0 bootstrap. User can edit before pipeline starts.

```json
{
  "idea": "",
  "mode": "new",
  "techStack": "auto",
  "maxIterations": 3,
  "outputDir": ".founder",
  "gates": {
    "postEval": "pause",
    "postImplementation": "pause",
    "postVerification": "pause"
  },
  "phases": {
    "eval": true,
    "architecture": true,
    "tdd": true,
    "implementation": true,
    "e2e": true,
    "verification": true,
    "postRunAnalysis": true,
    "gtm": true
  },
  "tdd": {
    "strictness": "unit+integration",
    "coverageThreshold": 80
  },
  "gtm": {
    "readme": true,
    "pitch": true,
    "tweetThread": true,
    "productHunt": true,
    "changelog": true
  },
  "acceptanceCriteria": []
}
```

Gate options: `"pause"` (print results, wait for input), `"auto"` (auto-continue unless blocking issue), `"skip"` (no output, no pause).

Mode auto-detection: paths/directories resolve to `"existing"`, GitHub URLs clone first then `"existing"`, everything else is `"new"`.

Tech stack resolution: When `"auto"`, new ideas get a recommendation in Phase 1 and finalization in Phase 2. Existing projects auto-detect from manifest files in Phase 0.
